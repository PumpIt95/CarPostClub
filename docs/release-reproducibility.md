# Release reproducibility and audit remediation

This repository is the source of truth for CPC2 releases and local operational helpers. A release must be built from a clean reviewed commit; a developer checkout with tracked or untracked drift must never be used as the automation runner's working tree.

## Required release gate

Before a release is promoted:

1. `git status --porcelain` is empty.
2. `npm ci` succeeds.
3. `npm audit --omit=dev` reports no known production vulnerabilities.
4. `npm run check:syntax` succeeds.
5. `npm test` succeeds.
6. `PLAYWRIGHT_WORKERS=1 npm run test:e2e` succeeds.
7. `node scripts/build_release_manifest.mjs --source-commit "$(git rev-parse HEAD)"` produces a manifest containing the exact source commit.
8. The deployed health response reports the same source commit as the release manifest.

The release archive must be generated by `ops/deploy-production.sh`, which refuses a dirty worktree and archives `HEAD`. Do not copy files directly from the user's working directory to production.

## Runtime artifact policy

Automation run data, lock evidence, temporary inventory exports, and browser artifacts belong in the runtime state root, not in the source tree. They are ignored by `.gitignore` and must not be added with `git add -f`. Keep a bounded evidence retention policy and preserve only the proofs needed for incident reconstruction.

## External controls still required

The following cannot be safely enabled by a repository commit alone and require a reviewed GitHub/host change:

- protect `main` with required review and required CI;
- enable Dependabot, secret scanning, and CodeQL results in repository settings;
- publish the reviewed remediation branch and deploy its immutable commit;
- rotate any credential whose old value was present in the watchdog environment;
- perform the offsite backup/restore drill;
- apply production container limits, log rotation, and host patching;
- reconcile Facebook live-listing evidence before any publish or sold mutation.
